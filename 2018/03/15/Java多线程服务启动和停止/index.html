2.1 线程模型

线程真实的表现在OS当中，它是OS任务调度的基本单位。当OS选择让哪个线程执行的时候，就会将相应的资源（CPU、IO）调配给它。那么既然OS将资源给它了，那咱就不能让它白费电，所以通常我们会给它分配任务，让它利用OS的资源将任务执行完毕。

Java很好的将线程和任务做了分离，这一小节，我们先介绍线程模型。

2.1.1 单线程

优点：不用考虑临界资源、锁等多线程活跃性带来的麻烦
缺点：吞吐和响应都很差、系统资源利用率低

2.1.2 一个任务一个线程

优点：吞吐和响应较3.1提升、资源利用率高
缺点：需要考虑多线程活跃性、线程创建销毁频繁带来的资源消耗大、线程过多引起线程切换、可能导致系统崩溃

2.1.3 线程池

优点：根据OS和物理机实际条件调优线程池线程个数、吞吐和响应好、不会频繁创建线程、不会由于任务过多导致线程太多系统崩溃
缺点：需要考虑线程活跃性

2.1.4 结论

对比以上三种的优缺点，找到任务类型，选择合适的线程模型。比如，强烈要求顺序性的任务，我们会选择类似于第一种的方案，保证顺序性；比如互相独立的任务，我们会选择类似第三种的方案，保证高吞吐，低延时。

2.2 Executor 线程框架

Executor将任务和执行任务的线程分离开来。线程池当中的线程就像工人一样，不断从传送带上提取任务，处理完之后，从传送带上获取下一个任务，循环往复。线程池细节很多，本小节对线程池创建、任务执行、任务提交方式进行简单介绍。并不是所有的线程池都采用这种方式创建，如果用到了定制过的线程池还需谨慎。

2.2.1 框架图说明

这个线程池构造函数是参数最全的。

  • corePoolSize : 线程池中线程的个数
