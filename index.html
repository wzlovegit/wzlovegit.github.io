场景

Thread Local

Posted on 2019-11-09 | In base

相比于全局变量，ThreadLocal 并不是一个非常常用的技术，理解和使用起来比较生疏。但在有些 CPU 密集型的服务中，我们可能需要关注线程的负载情况，做一些降级和限流，以保证在极端流量的情况下，保证核心业务的运行。那么 ThreadLocal 或许是比全局变量更加合理的选择。一来编码直观，二来没有锁带来的开销。

ThreadLocal 直观理解

抛开 Java 对 ThreadLocal 实现的代码，从直观上，ThreadLocal 想要完成单个线程内的全局状态共享，想必是每个线程对应了一块只由自己可见的内存。比如有 T1 和 T2 两个线程：

T1 -> 只有 T1 可见的内存1
T2 -> 只有 T2 可见的内存2

T1 和 T2 均不可见对方的那一块内存。这样线程就可以将自己相关的上下文都存储到这一篇内存当中，由于其他线程不可见，也就不用担心多线程问题。

ThreadLocal 原理浅析

ThreadLocal 之所以不好理解，我觉得是因为我们平时习惯了面向对象的思维方式。面向对象编程中，绝大多数情况下是对象维持了状态，这个状态不管到哪个线程要用，它都是一样的。比如，有一个对象 Student student = new Student(name, age); 那么不管哪个线程读到 student，它都是这一个对象。这个对象对应的那一片内存不会换。student.getName(); student.getAge() 都是从这一片内存上读取一个值，在哪个线程都一样。理解起来比较直观，就像人的祖籍，走到哪里，me.getHome() 都是河北。

ThreadLocal 对象多了一层逻辑。虽然在每个线程中 ThreadLocal 对象都是同一个对象，但是它们在每个线程中映射的内存都是线程独有的一块内存。也不难理解，就和社保一样。我是河北人，在西安工作的时候，社保就交到了西安，现在在北京工作，那么社保就交到了北京。所以在西安这个线程中访问我的社保，那就是西安的社保；在北京这个线程中访问我的社保，那就是北京的社保。互不干涉，各管各的。

ThreadLocal 源码的分析已经有很多了，这里不做过多分析。只需把握住两点：

  1. 每个线程单独一片内存，getMap(t)，t 表示线程，可以获取到这片内存
  2. 这一片内存中存储了以 ThreadLocal 变量为 key，以 threadLocal.set(value) 的 value 为值的 Map。

所以在每个线程中读到的内容都不同。

内存泄露

Java 中有容器的地方，使用不当就会造成内存泄露。 ThreadLocal 的底层实现当中就有一个 Map，那么使用不当，引起内存泄露也不是什么难事。

由上图可以看到，一旦执行了 threadLocal.set(value) 方法，在线程中就会存储一个 threadLocal -> value 的映射。如果线程中直接将 threadLocalRef 置为 null，或者指向另外的 threadLocal 对象，那么堆里面当前的 ThreadLocal 对象可以被 gc，那么 Map 中的 key 就指向了一片被 gc 掉的内存，此时 Map 就不能通过这个 key 索引到这个 value，就会造成内存泄露。

使用心得

最近工作中用到了 ThreadLocal。看了一些案例，最常见的用法是将 ThreadLocal 实现成 static 的，这样就可以在多个线程中用同一个 ThreadLocal 对象访问到线程独有的内存。

但并非 ThreadLocal 就得是 static 的，需要根据场景而定。只需明白一个 threadLocal 变量对应 Map 中的一个映射关系。比如我把 ThreadLocal 作为一个非 static 的成员变量封到一个 AnyClass 中，那么一个 AnyClass 对象对会有一个 ThreadLocal 变量对应到 Map 中。

netty 3 之 React 模型

Posted on 2019-05-02 | In base

React 模型翻译为反应堆模型。这种用抽象名词解释抽象名词的方式真是让人崩溃，所以大家姑且不需要管这些。

上文我们简单提到了多路复用 IO 模型，由于多路复用 IO 模型是现今服务器主流的网络 IO 型，并且它也是 React 模型的重要组成部分，所以我们这节就详细介绍一下多路复用 IO 模型。

多路复用 IO

我不打算采用抽象的句子解释什么是多路复用，先从一个现实生活中的实例（其实操作系统以及各种业务系统的资源配置策略都来源于生活）来入手了解什么是多路复用。
我们去饭店吃饭，每次进去会有服务员将菜单提供给我们，并等待我们点餐，这就是典型的多路复用 IO。每个客人是一个客户端，饭店是提供做熟饭服务的服务端，下单的服务员是多路复用器，厨师是服务器的计算能力。

  • 一个下单的服务员每天会为非常多的客人下单，但同一时刻只为一个客人下单，下单完成以后，接着为后续的客人下单。服务器的一个 IO 线程监听非常多的 IO 读写事件，处理完当前 IO 事件以后，则继续处理后续的 IO 事件，这就是多路复用，内核提供了多路复用监听 API，即 select, kqueue, epoll 这些。
  • 下单服务员不会在下单完成以后自己跑到后厨做饭，而是将单子转交给后厨以后，自己继续为其他客人下单。服务器的 IO 线程通常也不会处理比较复杂耗时的业务，这会引起 IO 线程吞吐效率变低，导致响应客户端时间过长，用户体验差。IO 线程会将事件简单处理后转交给后面的工作线程。
  • 节假日的时候，一个下单服务员忙不过来了，怎么办呢？很简单，多几个人去下单就可以了。当用户接入量大，并发请求过多，一个 IO 线程忙不过来了，怎么办？也一样，多几个 IO 线程就可以了。
  • 当然如果人爆满，那对不起，只能在外面排队了。 服务器也一样，有它的承载上限，如果客户端同时发起请求并且超出了服务器的处理能力，如果采用先用先出原则，后面的响应时间将会变得非常长。比如大家在抢购的时候，就经常出现页面一直转圈的反应。

这里下单服务员并非 accept 线程， accept 其实是门口招呼大家进店的人员，招呼进来后，他会把你安排给下单服务员，然后你和他就没有关系了。

React 模型

下单服务员和后厨工作人员统一构成了 React 模型。在 Netty 中我们可以灵活配置 worker 线程组中线程的数量，其实就是配置 React 模型中多路复用器的数量，即配置下单服务员的数量。做饭是比较耗时的任务，不会让下单服务员去做饭，所以会指定另外的线程组去做饭。

  • 多路复用在 Netty 中的体现
  • 分离线程组在 Netty 中的体现

addLast 指定做饭线程组，invoker 中携带了执行 handlers 使用的线程组。

小结

多路复用就是多个 IO 共用一个 IO 线程的意思。 React 是在多路复用的基础上加了一些线程组的管理。

netty 2 之 IO 模型

Posted on 2019-04-30 | In base

我们知道，从操作系统的角度来看，IO 是其必不可少的一部分。网络上的数据流入到系统中为 Input 的一种，数据从系统中流出到网络上为 Output 的一种。那么我们应该如何高效的处理网络 IO 呢？

BIO

BIO Blocked-IO 的简写，即阻塞式 IO。阻塞是指应用程序在调用操作系统提供的系统 API（比如，read, write）的时候，会挂起当前进程或线程，直到该 API 返回结果。拿 TCP socket 编程来举例：

应用程序调用操作系统提供的 write API 将 “hello world” 写入内核缓冲区，内核将数据通过 socket 发送到对端内核缓冲区，如果由于某种原因字母 ‘o’ 丢失，那么 TCP 会触发发送端重发 ‘o’ 到对端，但此时对端的 read 方法将一直阻塞，直到字母 ‘o’ 收到以后，才会从内核返回数据给上层应用，此时 read 才能返回，并继续执行后续的内容。

write read 等用到阻塞式的 socket 上，就会一直等到 socket 返回才会执行后续指令。在现实生活中如果有人如此，你会觉得他比较“轴”，比如，烧热水的时候，他一直盯着壶直到水开才会干别的事情。

BIO 的缺点就是阻塞进程或线程。现在服务器动则成千上万的用户接入，如果采用 BIO 的形式，为了快速响应用户，就需要成千上万的用户进程或线程，这种会给操作系统对进程或线程的调度带来很大的负担。我们可以观察到的现象是：CPU 满负荷运行，然而用户那边得到的反馈是响应太慢。
BIO 的优点是编码简单。BIO 常见于客户端开发，因为客户端通常和服务器保有一个连接，它发送了请求之后，如果服务器没有响应，那么它只能等待。比如 Jedis 采用了 BIO 开发。当然现在也不断出现很多 NIO 开发的客户端，这个在后面讨论。

Non-blocked IO

与 BIO 对应的就是 Non-blocked IO，即非阻塞式 IO。非阻塞式 IO 是指应用程序在调用操作系统提供的系统 API（比如 read, write）的时候，如果操作系统数据已经准备好，则直接返回数据，否则返回失败。read 和 write 操作不会导致进程或者线程挂起进入阻塞状态。

与 BIO 类似，数据会通过 socket 经由协议栈通过光纤等介质传输到对端。与 BIO 不同的是，非阻塞式 IO 在操作系统数据未准备好之前，会直接返回失败，而不是将进程或线程设置为阻塞状态。

write read 等用到非阻塞 socket 上，就会立即返回。比如，烧热水的时候，如果水没开，则立即去干其他事情，过一会儿再回来看一下是否开了。

目前内核提供了多路复用器（select，epoll，kqueue）,由于 socket 并不是一直有数据进行传输，或者由于网络的不稳定因素，会出现内核数据暂时不可读的情况，所以我们通常在多路复用器上监听多个非阻塞式 IO。一旦该 IO 可读或者可写，我们再进行读取或者写入。反应堆模型是目前最常见的服务器 IO 模型，Netty 就是采用了这一模型，利用这一模型我们可以灵活配置 IO 线程和业务线程的数据，充分使用计算机资源。这个后续专门讲解。

AIO

AIO Asynchronized-IO 的简写，即异步 IO。不管是 BIO 还是 Non-blocked IO，都是同步 IO。这里的同步和异步是针对用户和内核所说的。对于同步 IO，是指 socket 是否可读或可写，是需用用户进程不断地去扫描查看才能知道的；对于异步 IO，是指 socket 在可读或可写，是内核知道以后，反过来通知用户进程的（具体怎么通知，我也没试过，目前这个不是我们关注的重点，内核可以通过回调、信号等等机制去处理 socket 的可读可写事件）。看过几家客户端和服务器代码，目前没见过 AIO 的应用场景，暂时知道有这么个概念即可。

还是采用上面的现实实例：水烧开后，壶发出蜂鸣声，就是异步，因为不用我们一直去看水是否开了。

小结

虽然很多人把多路复用 IO 列为单独的一种 IO，但我还是把它划分到了 Non-blocked IO，因为我觉得这两个在 IO 模型上没有太大区别，只是为了提升性能，内核帮我们做了一些事情。

对于 Java 的 NIO，是 Java 处理 IO 的一种改进。在 NIO 之前， Java 多采用流式方式而非 buffer 方式处理 IO，并且 Java 大多是提供单通道（比如 InputStream 只有读取的能力， OutputStream 只有写入的能力）能力而非双通道（比如 Channel）的方式处理数据。大家容易把 NIO 和 Non-blocked IO 混在一起谈论，是因为 NIO 在网络库上使用广泛，而网络库目前大多基于多路复用 IO 模型。

netty 1 之 netty 简介

Posted on 2019-02-27 | In base

开发面临的问题

  • 微服务架构现已广泛应用于商用系统，然而微服务之间并非完全隔离的两套系统，它们之间的通信在所难免。如何快速实现一个具有可维护性、高性能的 RPC 框架？
  • 常用协议通常具有自己的特定场景，比如 HTTP 通常用于超文本传输，而不会用于邮件传输、大文件传输和实时媒体传输。为了更好的支撑自己的现有业务，有些情况下我们会自建协议服务器，那么如何快速实现具有可维护性、高性能的自建协议服务器呢？

由此可知，自建或扩充和业务比较贴合的协议是非常常见的业务场景。我们该如何去快速开发一个协议服务器，并且达到可扩展、高性能、稳定的要求？Netty 是一个不错的选择。

Netty 简介

Netty 是一个基于事件驱动的 Java 网络应用框架。

我们知道，应用层数据在由 A 发送到 B 的过程，数据需要依次经过传输层、网络层、数据链路层、物理层才会通过光纤等介质发送到 B。B的物理层收到以后，一次经过数据链路层、网络层、传输层的处理以后，才会由内核反馈给应用层。至于每一层的功能，有需要的可以翻一下教材。

  • 事件驱动

    事件驱动是指，内核向应用层上报网络 IO 事件以后，应用层进程或线程从 parking 或 waiting 状态进入 running 状态开始处理事件。

  • 网络框架

    Netty 基于传输层 socket 完成事件驱动模型的设计，将复杂繁琐且几乎不变的 socket 通信封装起来，给应用层提供更加便捷的上层视图，便于开发人员专注自定义协议开发。

点赞 Hystrix 设计

Posted on 2019-02-24 | In base

通过 ribbon介绍，我们了解到 Ribbon 是一个 HTTP 负载均衡客户端，但只有负载均衡并不能达到商用系统的要求，本文我们一边分析为什么，一边了解 Hystrix 提供的解决方案。

降级、熔断、限流是我们经常听到的概念，也是商用系统稳定运行的关键。降级、熔断发生在服务请求端，限流通常发生在服务提供端。

降级

服务 A 请求服务 B，由于网络抖动，请求发送失败，此次 HTTP 请求失败。如果不做处理，则会将异常抛到用户手中，用户直接一个黑人问号脸：

懂一点的用户觉得这个软件很 low，不懂的觉得不会是病毒吧。此时作为一个码农，你可能会说，都失败了，我有什么办法？网络异常，怪我喽？

答案是：有办法，确实怪你考虑得不周全。网络异常是不可避免的，因此 CAP 里面中有了 AP 和 CP 的选择，我们的方案必须考虑网路抖动。此时给用户提供一个错误引导比返回一串不知所云的异常要好得多，比如：”请查看网络是否正常，请重试 … “。

这就是我们所说的降级。虽然网络出现了暂时性异常，但我可以选择返回一个用户能理解的失败，而非返回一串异常。 值得一提的是 Hystrix 能够帮我们包装 HTTP 请求，捕获请求过程中的异常，并提供 Fallback 方法让我们实现降级逻辑。

熔断

服务 A 请求服务 B，如果服务 B 的 b1 实例由于某种不可抗因素出现了较长时间的不可用。如果不做处理，就会出现概率性用户请求失败，用户会倾向于怀疑人品。

作为一个码农，你可能会说，不可抗因素，让运维上啊。运维上不假，但从问题发生到反馈到运营部再到研发定位再到运维下掉 b1 实例，修复完成。如果发生在后半夜，可能会持续很长时间。这么多的服务实例，随便一个故障都触发这么长时间的一个人力流程，用户体验无从谈起。在这种高压力状态下，大家除了让老板看着很努力，没有解决任何实质问题。

Hystrix 提供了熔断器。

  • 可以采用滚筒采样，周期计算的方式来判断实例是否正常。
  • 如果发现一个时间窗口内，某实例的请求失败率超过了阈值（当然采样个数要足够，不能采样一个失败，直接按 100% 失败处理），则触发熔断器打开，此时异常服务就被标记为了下线实例，不再使用
  • 熔断器打开后，后续请求直接进入 Fallback（通常是这样，如果配置了并发控制的话，则会快速失败抛异常，而不进入 Fallback）
  • 另外，在熔断器打开期间，一个 TEST 时间窗口，客户端会向标记为熔断的异常实例发送一次请求，如果还是失败，则继续打开熔断器，如果成功，则关闭熔断器，自动回复实例。

通过这种自动化的方式，让我们可以睡个好觉，上班以后，再根据监控，处理这个 b1 实例。

超时和重试

服务 A 请求服务 B，如果请求失败，A 就告诉调用者，你再试一下。就好像，你去政府部门办事，跟柜台小姐姐说明情况之后，她去找领导，领导正在开会，此时她告诉你，你过 10 min 再过来办一次（毫无违和感）。你略有不爽。后面你又来了 10 次，都是领导在开会。

后来有了位有脑子的领导，发起了一次行政效能革命。你只需要把材料交个柜台小姐姐回家就好。小姐姐去给你找到领导，把章盖好，是不是很好。这才是为人民服务，人民拥护的好政策。

服务 A 在一次请求失败后，需要重试请求服务 B。Ribbon 自身也带有重试机制控制。

用 Spring Cloud 做微服务，难免需要发起 HTTP 请求来完成服务间通信。HTTP 请求是基于 TCP 的 7 层协议。使用 HTTP 长连接代替短连接（减轻 TCP 建链、断链损耗）是我们最先想到的提高性能的方法。很多人会提到，异步化的，把业务线程和IO线程分隔开。异步化是对的，不过他们少了解了一个内容，就是 HTTP 请求需要 req/rsp 才算完成，才能在长连接发送另一次 req/rsp。所以很多人考虑得方案如下：

服务保活

Posted on 2019-02-18 | In framework

服务保活是微服务治理的基本要求，服务注册、服务摘除的自动化都离不开服务保活。那么如何去保活服务呢？

大部分软件工程师是做业务的，但是业务是繁荣的，Not only web！我们通常会依赖相关领域中重要的三方服务，比如通信领域的 FreeSwitch。FreeSwitch 并没有原生的集群能力，比如选主、failover等。针对这一类服务，我们需要了解如何保活服务。

续约 VS 探测

续约是将注册中心作为服务，三方服务作为客户端，周期性向注册中心上报自身状态，一旦注册中心连续多次无法获取三方服务的续约请求，则认为三方服务出现故障。如下图所示：

探测是将注册中心作为客户端，三方服务作为服务，注册中心周期性发送 PING 等待三方服务返回 PONG，一旦注册中心连续多次无法获取三方服务的 PONG 回应，则认为三方服务出现故障。如下图所示：

这两种方式选择任意一种均可以满足要求。

连接 VS 服务

接触过连接池的工程师都知道连接池中的连接通常也需要保活，也会采用 PING / PONG 方式，比如 Jedis。但本人认为连接的保活和服务的保活是两个不同层次的内容，一个连接的丢失并不能象征服务故障。

连接的保活，是确保在连接上可以传递信息。服务的保活，是确保服务可用。服务的保活 PING/PONG 可以通过任一连接发送，只要成功即可认为服务可用。

结论：连接的保活放到底层连接实现。服务的保活放到服务定时任务中采用 PING/PONG 实现。

pool 池化技术

Posted on 2019-02-15 | In base

近期在优化 FreeSwitch ESL 客户端相关的业务实现的过程中，对 ESL 提供的 Client 进行了封装，设计并开发了 FreeSwitch 连接池，命名为 JreeSwitch（学习 Jedis 命名方式）。本文围绕如何设计池和如何使用池展开讨论，对池化技术的精髓进行介绍。

对象池

对象池是统一管理多个对象生命周期的一种技术。对象池最大的优点是省去频繁创建和销毁对象的性能损耗，其次可以将对象生命周期的管理复杂度集中管理，减少开发和运维难度。

小贴士： 池化技术在现实生活当中随处可见，即共享技术。例如共享单车属于单车资源池化，高铁属于交通资源池化，我们不创建共享单车和高铁，但是我们可以方便得使用它们。

用处

正是由于对象池的这些优点，使得我们经常见到线程池、连接池等名词。既然对象池如此优秀，为什么不讲所有的对象都放到对象池呢？我总结出如下用法：

  • 对象一次创建和销毁的开销相对业务本身较大，通常涉及内核资源的调度（系统 API 调用），比如连接、线程、内存等
  • 对象可以复用，并非临时性对象，比如连接创建之后，在连接上发送命令即可，无需频繁建链。

对象池设计

参照 GenericObjectPool 进行讲解

我们在使用对象池时是不关注对象何时创建的，用完对象之后，同样也不关注对象何时被销毁，那么对于一个用于租赁对象的对象池来讲，它需要具备以下几个核心能力。

  • borrowObject : 对象池出租对象给用户
  • returnObject : 用户归还对象给对象池
  • 检查对象状态 : 出租前、归还后检查
  • 对象池检查 : 定时检查，根据对象状态和对象池状态决定是否回收或创建连接
  • 动态扩容缩容 : 根据对象池状态，动态扩充新对象或销毁对象

GenericObject

入参为 PooledObjectFactory 和 GenericObjectPoolConfig。其中 PooledObjectFactory 用于提供对象的创建、激活、去激活、销毁等方法给 GenericObjectPool 管理对象生命周期时使用。GenericObjectPoolConfig 用于完成对象池的配置，比如最小空闲对象数等。

PooledObjectFactory

  • makeObject()
    创建对象实例。 当我们调用对象池的 borrowObject 方法获取对象时，在规则满足的情况下（比如池中无空闲对象并且没有到达最大对象数），那么对象池会调用该方法生成一个新的对象放入对象池。

  • activateObject(org.apache.commons.pool2.PooledObject) 激活对象实例。当我们从对象池中 borrowObject 的时，会执行该方法先激活对象再返回给用户使用。

  • validateObject(org.apache.commons.pool2.PooledObject) 判断对象合法性。在对象池返回某个对象或者将对象归还到对象池中的时，若满足配置规则，则会调用该方法判断对象是否可用。

  • passivateObject(org.apache.commons.pool2.PooledObject) 去激活对象实例。当我们 returnObject 给对象池时，会调用该方法去激活对象。

  • destroyObject(org.apache.commons.pool2.PooledObject) 销毁对象。当定时检查对象状态和对象池状态时，若满足规则，则销毁一部分对象。

GenericObjectPoolConfig

管理对象池的最大个数，最大空闲对象个数和最少空闲个数。

快速实现

在 GenericObjectPool 的基础上，我们可以快速完成对象池化。

  • 实现 PooledObjectFactory ：提供对象的创建、激活、去激活、销毁等方法。
  • 创建 GenericObjectPool ： 完成对象池生命周期管理（当然可以通过 GenericObjectPoolConfig 设置对象池的大小）。

使用者在使用时无需关注何时创建、何时销毁等操作就可以复用以前的对象，非常方便。
